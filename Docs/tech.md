# Technology Stack

## 개발 기조

이 프로젝트는 **원소스 멀티플랫폼(OSMP)** 전략을 따른다. 하나의 Godot 프로젝트로 PC 웹과 모바일 앱/웹 플레이를 동시에 대응하며, 초기 개발에서는 빠른 구현과 테스트를 우선한다.

핵심 리스크는 실시간 멀티플레이 동기화, 모바일/데스크탑 조작감 차이, 3D 에셋 호환성이다. 따라서 화면은 **가로 화면(Landscape)** 기준으로 고정하고, 이동 조작은 **터치/클릭 기반 방향 이동**으로 통일한다.

## 기술 스택

- **게임 엔진:** Godot Engine
- **클라이언트 언어:** GDScript
- **개발 IDE:** VS Code + Godot 디버거
- **실시간 서버:** Node.js
- **통신:** 순수 WebSocket, Node.js `ws` 라이브러리
- **데이터 저장:** MVP 단계에서는 DB 미도입, 서버 인메모리 상태 관리
- **웹 빌드 배포:** GitHub Pages 또는 Vercel
- **서버 호스팅:** Render 또는 AWS EC2 Free Tier

## 클라이언트 규칙

- Godot 씬은 역할별로 작게 나눈다: 로비, 게임 맵, 플레이어, 새끼오리, UI, 네트워크 매니저.
- 모든 플레이어 입력은 클라이언트에서 즉시 반응시키되, 중요한 판정은 서버 상태를 기준으로 보정한다.
- 모바일과 데스크탑 모두 같은 게임 화면을 사용한다.
- 조작 UI는 가로 화면 기준으로 설계한다.
- 버튼과 텍스트는 모바일에서도 겹치지 않도록 안전 영역을 고려한다.

## 서버 규칙

- 서버는 방(Room) 단위로 플레이어, 새끼오리, 타이머, 점수, 감옥 상태를 관리한다.
- 위치 동기화는 일정 주기로 브로드캐스트한다.
- 새끼오리 스폰, 획득, 반납, 잡기, 감옥 이동, 게임 종료 같은 핵심 판정은 서버가 최종 결정한다.
- MVP에서는 서버 재시작 시 방 데이터가 사라져도 허용한다.
- Colyseus는 상태 동기화가 복잡해질 때만 도입을 검토한다. 초기 구현은 `ws`로 시작한다.

## 에셋 규칙

- 직접 제작보다 무료 3D 에셋 조합을 우선한다.
- Godot 호환성을 위해 `.gltf` 또는 `.glb` 포맷을 우선 선택한다.
- 캐릭터는 가능하면 걷기/수영 애니메이션과 리깅이 포함된 모델을 사용한다.
- 배경과 지형은 Kenney `Nature Kit`, `Water Craft Kit` 같은 로우폴리 에셋을 우선 검토한다.
- 캐릭터는 Sketchfab, Poly Pizza 등에서 라이선스와 출처 표기 조건을 확인한 뒤 사용한다.

## 성능 기준

- 목표 인원은 한 방 기준 2~6명으로 시작한다.
- 위치 전송 주기는 너무 촘촘하게 잡지 않고, 보간으로 움직임을 부드럽게 만든다.
- 모바일 브라우저에서도 실행 가능하도록 폴리곤 수와 이펙트를 제한한다.
- 네트워크 지연이 있어도 게임 규칙이 깨지지 않도록 서버 권한을 유지한다.

## 추후 검토

- **Redis:** 방 상태 보존, 여러 서버 인스턴스, 전역 랭킹이 필요할 때 검토한다.
- **Colyseus:** `ws` 기반 직접 동기화가 복잡해질 때 검토한다.
- **로그인/DB:** 플레이 기록, 캐릭터 해금, 랭킹이 필요할 때 추가한다.
