## 기술 스택

### 💻 기술 스택 (전체 요약)

1. **개발 기조:** 하나의 코드로 PC 웹과 모바일 앱을 동시 대응하는 **원소스 멀티플랫폼(OSMP)** 전략 채택
2. **생산성 극대화:** 클로드(Claude)의 코드 생성 효율을 높이기 위해 컴파일 과정이 없고 가벼운 스택 위주로 구성
3. **리스크 관리:** 플랫폼 간 UI 깨짐 및 조작감 저하를 방지하기 위해 **'가로 화면(Landscape)' 고정** 및 **'터치/클릭 방향 이동'으로 조작계 통일**

    - **1. Kenney (케니) — 배경/지형용 (추천)**

        - **단점:** 단순 도형 위주라 디테일한 동물 캐릭터는 부족함.
    - **2. Sketchfab (스케치팹) — 캐릭터용 (Downloadable/CC 필터 필수)**
        - **용도:** 오리, 거위, 수달 등 주요 크리처 에셋 확보
        - **장점:** 수영·걷기 등 애니메이션(Rigging)이 포함된 에셋이 많아 구현이 편함.
        - **단점:** 화풍이 제각각이라 이질감이 들 수 있고, 출처 표기 의무 확인 필요.
    - **3. Poly Pizza (폴리 피자) — 소품용**
        - **용도:** 연잎, 물고기 등 단품 오브젝트 검색용
        - **주의:** 크기(Scale) 및 중심점(Pivot) 보정 작업 필요.
    - **🚨 필수 체크 리스크**
        1. **포맷:** 호환성 세팅 시간 절약을 위해 **`.gltf` 또는 `.glb` 확장자**만 우선 선택.
        2. **리깅:** 움직임 구현을 위해 정지형 모델이 아닌 **애니메이션 내장 모델**인지 필히 확인.

    ## 🛠️ 냉정한 리스크 관리 가이드 (몰입캠프 팁)
    
    3D 에셋을 다운로드할 때 피해야 할 **현실적인 함정** 두 가지만 기억하세요.
    
    1. **`.fbx` 대신 `.gltf`나 `.glb` 파일 우선 선택하기**
        - Godot 엔진은 **`.gltf` / `.glb`** 포맷을 가장 완벽하게 지원합니다. Blender(블렌더) 같은 3D 툴을 거치지 않고 바로 Godot에 집어넣으려면 확장자를 반드시 확인하세요. C++만큼이나 3D 포맷 호환성 세팅도 시간을 많이 잡아먹습니다.
    2. **애니메이션 뼈대(Rigging) 유무 확인**
        - 오리가 발을 저으며 수영하는 연출을 하려면 에셋 자체에 애니메이션(Rig)이 포함되어 있어야 합니다. 만약 정지된 3D 덩어리(Static Mesh)라면, 클로드에게 코드를 짜달라고 해도 오리가 인형처럼 뻣뻣하게 미끄러지듯 움직일 수밖에 없습니다.
- 엔진
    
    ### 🎮 엔진 및 언어
    
    - **엔진:** `Godot Engine (고도 엔진)` (가볍고 WebGL 및 모바일 내보내기 성능 우수)
    - **언어:** `GDScript (지디스크립트)`
        - 별도의 컴파일 과정이 없어 수정 후 즉시 확인 가능 (생산성 루프 단축)
        - 클로드(Claude)가 가장 오류 없이 고품질로 짜주는 언어 세팅
        - 가벼운 모바일 앱 용량 및 웹(WebGL) 원클릭 빌드 지원
    - **IDE (개발 환경):** `VS Code (Visual Studio Code)`
        - 클로드의 코드를 복사·붙여넣기하고 다중 파일을 관리하기에 내장 에디터보다 압도적으로 유리
        - 단, 에러 로그 확인 및 실행(F5)은 Godot 엔진 디버거 활용
- 벡엔드
    
    ### 🌐 벡엔드
    
    - **기술 스택:** `Node.js (노드제이에스)`
        - **선택 이유:** 싱글 스레드 이벤트 루프 기반이므로, 실시간 멀티플레이 서버 구현 시 자바(Spring) 등에서 발생하는 복잡한 '스레드 동기화(Lock)' 문제를 신경 쓰지 않아도 됨.
        - 자바스크립트 파일 단위로 가볍게 구조화할 수 있어 클로드에게 서버 코드를 요청하고 반영하는 속도가 스프링 대비 배 이상 빠름.
    - **대안 검토:** `Colyseus (콜리세우스) 프레임워크`
        - 실시간 오브젝트(새끼 오리 동기화 등) 상태 관리가 지나치게 복잡해질 경우, 고도 엔진 전용 플러그인을 지원하는 콜리세우스 도입을 검토함.
        
- DB/캐시
    
    ### 🗄️ DB/캐시
    
    - **기조:** 개발 기간 단축 및 AI 코딩 효율을 위해 프로토타입 단계에서는 무리한 DB 도입을 배제하고 인메모리 방식으로 우선 구현
    - **선택 스택:** `미도입 (Node.js 인메모리 관리)`
        - **이유:** 현재 동시 접속 인원이 소수이고 유저 로그인 기능이 없으므로, Node.js 서버 내부의 변수(예: `const rooms = {}`)와 자바스크립트 메모리만으로도 데이터가 렉 없이 완벽히 처리됨.
        - 불필요한 환경 세팅 시간을 줄여 클로드(Claude)와의 핵심 로직 구현에 집중.
    - **추후 검토 스택:** `Redis (레디스)`
        - **검토 조건:** 개발 4단계(최적화 및 배포) 진입 시, **실시간 전역 랭킹 시스템**이나 **서버 재시작 시의 방/유저 데이터 보존**이 반드시 필요하다고 판단될 경우에만 추가 도입을 검토함.
- 웹소켓
    
    ### 🔌 웹소켓
    
    - **기술 스택:** `ws` 라이브러리 (순수 웹소켓)
        - Godot 엔진의 내장 `WebSocketPeer` 클래스와 추가적인 프로토콜 레이어(STOMP 등) 없이 가장 순수하게 100% 호환됨.
        - 클라이언트-서버 간 JSON 기반 데이터 송수신 구현이 직관적임.
        - *주의 리스크:* 위치 및 상태 동기화 로직(오리가 새끼 오리를 획득 시 다른 유저 화면에서도 사라지게 하는 판정 등)을 직접 구현해야 하므로 데이터 꼬임 버그 방지를 위해 타이트한 검증 필요.
